
# 07 · 탐지 룰 설명

[[00-프로젝트-계획서|← 계획서로 돌아가기]]

> 각 룰은 "무엇을 왜 위험하게 보는가"가 핵심이다. 코드는 `apkscan/rules.py`.

---

## 현재 구현된 룰

### PERM-001 · 위험 권한 사용
구글이 'dangerous'로 분류한 권한(SMS·위치·카메라·접근성·설치 등)을 요청하는지 검사.
특히 **접근성 서비스**나 **외부 APK 설치** 권한은 악성 앱의 단골 → 심각도 상향.

> 왜? 이런 권한 조합은 화면 탈취·인증번호 가로채기·드로퍼(다른 악성 앱 설치)의 신호.

### MANIFEST-001 · 디버그 모드 (`debuggable=true`)
릴리스 빌드에 디버그 플래그가 남으면 누구나 앱에 디버거를 붙일 수 있다. **HIGH**.

### MANIFEST-002 · 백업 허용 (`allowBackup=true`)
`adb backup`으로 앱 데이터가 유출될 수 있다. (기본값이 true라 흔함) **LOW**.

### MANIFEST-003 · 평문 트래픽 허용 (`usesCleartextTraffic=true`)
암호화 안 된 HTTP 통신 허용 → 중간자 공격 노출. **MEDIUM**.

### MANIFEST-004 · 노출된 ContentProvider
다른 앱이 데이터에 접근할 통로. 권한 보호 없이 exported면 위험. **MEDIUM**.

### MANIFEST-005 · 외부 노출 컴포넌트
exported activity/service/receiver = 다른 앱이 호출 가능한 **공격면**. **INFO**(참고용).

### CERT-001 · 디버그 인증서 서명
디버그 키는 모든 개발자가 공유하는 공개 키 → 사실상 서명이 없는 것. **HIGH**.

### CERT-002 · 약한 서명 알고리즘 (SHA-1)
SHA-1은 충돌 공격에 취약한 구식 해시. **LOW**.

### SECRET-* · 하드코딩된 비밀키
코드 문자열에서 AWS 키, Google API 키, Private Key, Slack 토큰, JWT,
Firebase URL 패턴을 정규식으로 탐지. **HIGH**.

> 왜? 키가 앱에 박혀 있으면 누구나 디컴파일해서 빼낼 수 있다.

## 위험 점수 계산

각 Finding의 심각도에 가중치를 매겨 합산 (최대 100):

| 심각도 | 가중치 |
|------|:---:|
| INFO | 0 |
| LOW | 5 |
| MEDIUM | 15 |
| HIGH | 30 |
| CRITICAL | 50 |

→ 로직은 `analyzer.py`의 `risk_score`.

---

## 룰 추가하는 법 (확장)

`rules.py`에 함수 하나 추가하고 `run_all`에 끼우면 끝:

```python
def check_my_rule(facts: ApkFacts) -> list[Finding]:
    out = []
    if <조건>:
        out.append(Finding(
            rule_id="MYRULE-001",
            title="...",
            severity=Severity.MEDIUM,
            detail="...",
            recommendation="...",
        ))
    return out
```

그리고 `run_all` 안에 `findings += check_my_rule(facts)` 한 줄.

> 이 구조 덕에 [[03-일정-로드맵]] 2주차의 "룰 10개로 확장"이 쉽다.

## 2주차에 추가할 룰 후보

- 위험 API 호출: `Runtime.exec()`, `DexClassLoader`(동적 코드 로딩), `SmsManager.sendTextMessage`
- 루팅 탐지 우회 / 에뮬레이터 탐지 문자열
- 과도한 권한 대비 실제 사용 불일치
- 오래된 `targetSdk` (최신 보안 정책 미적용)
