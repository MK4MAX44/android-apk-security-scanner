## 이 방식을 선택한 이유 (의사결정 배경)

처음에 4가지 방향을 놓고 고민했다:

| 방향 | 내용 | 선택? |
|------|------|:---:|
| A. N-day 취약점 분석 | 공개 CVE 역분석·PoC 재현 | ❌ |
| B. 퍼징 하네스 구축 | 자동 크래시 탐지 인프라 | ❌ |
| C. **방어 탐지 도구** | **APK 정적 분석기** | ✅ |
| D. 순수 리서치/보고서 | 문서 조사 정리 | ❌ |

**C를 고른 이유:**

1. **결과물이 확실하다** — A·B는 "운"이 개입한다(취약점을 못 찾을 수 있음).
   C는 코드를 짜면 무조건 동작하는 도구가 남는다.
2. **시작 문턱이 낮다** — 어셈블리·메모리 구조 없이 파이썬만으로 시작 가능.
   난이도가 낮은 데서 시작해 [[08-확장-아이디어]]로 천장을 올릴 수 있다.
3. **목적에 맞다** — 목적이 *포트폴리오 + 취업/자격증*이었다.
   동작하는 도구 + 리포트가 이 목적에 가장 직접적이다. → [[01-왜-이-프로젝트인가]]
4. **자연스러운 다음 단계가 있다** — 정적 분석에 익숙해지면
   동적 분석(Frida) → N-day 분석(방향 A)으로 넘어가기 쉽다.

> 요약: **"가장 안전하게 결과물이 나오면서, 나중에 더 어려운 길로 확장 가능한 입구"**

## 목표 (SMART)

- **S**pecific: APK를 입력받아 보안 위험을 탐지하는 CLI 도구
- **M**easurable: 탐지 룰 10개 이상, 텍스트/JSON/MD 출력, 실제 APK로 검증
- **A**chievable: 오픈소스 라이브러리(androguard) 위에서 구현
- **R**elevant: 모바일 앱 보안 실무·자격증 개념과 직결
- **T**ime-bound: 5~6주 → [[03-일정-로드맵]]

---

## 범위: 만드는 것 (In scope)

- ✅ 매니페스트 분석: 권한, debuggable, allowBackup, cleartext, exported 컴포넌트
- ✅ 서명 인증서 분석: 디버그 키·약한 알고리즘 탐지
- ✅ 코드 문자열 분석: URL 추출, 하드코딩 시크릿(API 키/토큰) 탐지
- ✅ 심각도 기반 위험 점수(0~100)
- ✅ 3종 리포트 출력 (사람용·기계용·문서용)
- ✅ 룰 기반 엔진 (룰을 데이터로 → 확장 쉬움) → [[07-탐지-룰-설명]]

## 범위: 안 만드는 것 (Out of scope, 최소 이 단계에선)

- ❌ 동적 분석 (앱 실행 후킹) — [[08-확장-아이디어]]에서 별도
- ❌ 자동 익스플로잇 생성
- ❌ 난독화 완전 해제 / 디컴파일 UI
- ❌ 실시간 안티바이러스

> 범위를 좁게 잡는 이유: **끝까지 완성되는 작은 것**이 미완성의 큰 것보다 낫다.

---

## 완료 기준 (Definition of Done)

- [ ] 임의의 APK에 대해 크래시 없이 리포트 생성
- [ ] 탐지 룰 10개 이상, 각 룰에 설명·권고 포함
- [ ] 정상 앱 vs 의도적으로 취약한 앱에서 결과가 유의미하게 다름
- [ ] README와 이 볼트 문서로 남이 따라 할 수 있음
- [ ] GitHub 공개
