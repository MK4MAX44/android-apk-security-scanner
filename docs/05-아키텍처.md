## 설계 원칙: 관심사 분리

도구는 4단계 파이프라인이다. 각 단계는 **한 가지만** 한다.

```
APK 파일
   │
   ▼
[1] 추출 (extractors.py)   ← 사실만 수집, 판단 안 함
   │   권한/매니페스트/인증서/문자열 → ApkFacts
   ▼
[2] 룰 (rules.py)          ← 사실을 보고 위험 판단
   │   ApkFacts → [Finding, Finding, ...]
   ▼
[3] 분석 (analyzer.py)     ← 위 둘을 엮고 위험 점수 계산
   │   → AnalysisResult
   ▼
[4] 리포트 (report.py)     ← 사람/기계/문서용 출력
       text / JSON / markdown
```

> **왜 이렇게 나눴나?**
> 룰을 추가할 때 추출·출력 코드를 건드릴 필요가 없다.
> 출력 형식을 바꿔도 분석 로직은 그대로다. → 확장·테스트가 쉬움.

---

## 파일 지도

| 파일 | 역할 | 핵심 |
|------|------|------|
| `apkscan/extractors.py` | 추출 | `ApkFacts` 데이터클래스, `extract_*` 함수 |
| `apkscan/rules.py` | 판단 | `Severity`, `Finding`, `check_*` 함수, `run_all` |
| `apkscan/analyzer.py` | 오케스트레이션 | `analyze()`, `AnalysisResult`, 위험 점수 |
| `apkscan/report.py` | 출력 | `to_text` / `to_json` / `to_markdown` |
| `apkscan/cli.py` | 진입점 | argparse, `python -m apkscan` |

---

## 핵심 자료구조

### `ApkFacts` — "이 APK에 대해 알아낸 사실"
권한 목록, debuggable 여부, exported 컴포넌트, 인증서 정보, 코드 문자열 등.
판단은 없다. 오직 관찰된 사실.

### `Finding` — "발견한 위험 신호 하나"
```python
Finding(
    rule_id="PERM-001",
    title="위험 권한 3개 사용",
    severity=Severity.HIGH,
    detail="...",
    recommendation="...",
)
```

### `Severity` — 심각도 (INFO < LOW < MEDIUM < HIGH < CRITICAL)
정렬·점수 계산에 쓰는 정수 enum.

---

## 데이터 흐름 예시

1. `analyze("app.apk")` 호출
2. androguard `AnalyzeAPK`가 APK를 해석 → 매니페스트 XML, DEX 코드
3. `extract_*`가 `ApkFacts` 채움
4. `run_all(facts)`가 모든 `check_*` 룰 실행 → `Finding` 리스트
5. `AnalysisResult`가 위험 점수 계산
6. `report.to_markdown()`이 옵시디언용 리포트 출력

---

## 왜 androguard인가

- **순수 파이썬** → 자바/aapt 설치 불필요 (이 환경엔 Java가 없음)
- APK 파싱·매니페스트 디코딩·DEX 분석·인증서까지 한 번에
- 정적 분석의 사실상 표준 라이브러리
