## 결과 요약

| 앱 | 위험 점수 | 발견 | 최고 심각도 |
|---|:---:|:---:|:---:|
| InsecureBankv2 (취약) | **85** | 5 | HIGH |
| F-Droid Basic (정상) | 65 | 4 | HIGH |

→ 취약 앱이 명확히 더 높은 점수. **도구가 둘을 구분한다.** ✅

---

## 무엇이 둘을 갈랐나

| 신호 | InsecureBankv2 | F-Droid |
|---|:---:|:---:|
| 디버그 모드(`debuggable=true`) | ✅ 있음 (HIGH) | 없음 |
| 위험 권한 개수 | 5개 (SMS·통화기록·연락처 등) | 3개 |
| SMS 발송 API | ✅ | 없음 |
| 동적 코드 로딩(DexClassLoader) | ✅ | 없음 |
| 낮은 targetSdk | 22 (오래됨) | 35 (최신) |

**핵심 차이:** 취약 앱은 `debuggable=true`(릴리스에 있으면 안 되는 것),
SMS 발송, 동적 코드 로딩, 오래된 SDK를 모두 갖고 있었다.
정상 앱에는 이 중 어느 것도 없었다.

---

## 오탐(False Positive) 분석 — 중요

점수만 보면 F-Droid의 65점도 낮지 않다. 왜일까?

F-Droid는 **앱스토어**라서 다음이 "정당하게" 필요하다:
- `REQUEST_INSTALL_PACKAGES` 권한 (앱 설치가 본업)
- `Runtime.exec` / `ProcessBuilder` API 호출

즉 이건 **탐지는 맞지만(True Positive), 맥락상 양성(benign)** 인 경우다.
→ 정적 분석의 근본 한계: **"무엇을 하는가"는 보이지만 "왜 하는가"는 모른다.**

> 교훈: 위험 점수는 **자동 판정이 아니라 "사람이 봐야 할 우선순위"** 로 써야 한다.

---

## 이 도구의 한계 (정직하게)

- **런타임을 못 본다** — 실제 실행 시 행위는 [[08-확장-아이디어]]의 동적 분석 몫.
- **난독화에 약하다** — 문자열/시크릿 탐지는 난독화되면 회피된다.
- **점수는 휴리스틱** — 절대적 안전/위험 판정이 아니다.
- **맥락을 모른다** — 위 F-Droid 사례처럼 정당한 사용도 걸린다.

---

## 정밀도를 높이려면 (다음 개선안)

1. **권한×API 조합 룰** — "SMS 권한 + SMS 발송 API + 인터넷" 같은 조합에 가중.
2. **앱 카테고리 인식** — 앱스토어/백신 등은 특정 권한을 예외 처리.
3. **더 많은 표본** — 정상 20 + 악성 20으로 오탐/미탐율을 수치화.
4. **시크릿 검증** — 엔트로피 계산으로 진짜 키와 예시 문자열 구분.
